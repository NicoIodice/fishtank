## 2026-06-02 — Coaching Session

### Users & Deployment Context

**Primary users (both first-class):**
1. **Backend/full-stack developer** — local dev machine; needs mocks so services run without live dependencies; switches mocks on/off to test edge cases
2. **QA/test engineer** — shared test environments (dev, qa, staging, uat); deploys for manual exploratory testing, automated system/API tests run by pipelines, and cross-team integration testing

**Key insight:** Config is environment-specific — dev mocks deliberately differ from test environment mocks. Volume-mounted config per deployment context is not just a technical choice, it's a user requirement.

**Priority:** Speed over team coordination. UI optimizes for individual efficiency (minimal clicks, fast feedback) not shared workspace management.

**Config mutation ownership:** In shared environments, only the QA engineer (or someone with their explicit consent) changes mocks or config. Pipelines are consumers, not mutators — they run tests against existing mocks, they do not promote or modify stubs. This is a deliberate, human-gated act.

### Authentication

- **Auth on by default** — long-lived API token, set once and persists
- **Disable via environment variable** — e.g. `FISHTANK_AUTH_ENABLED=false`; designed for local dev Docker overrides in a Dockerfile or docker-compose, not via config file
- **URLs are internal-network only** — no public exposure; org network boundary is the outer security perimeter

**Open question logged:** How is the default token generated/changed? Options: (a) generated on first run, printed to logs once; (b) set by the user as an env var (`FISHTANK_AUTH_TOKEN=...`); (c) both — auto-generated if not set, overridable. Needs a decision before implementation.

### JSON Config Role (FINAL — seeding + resync model)

The JSON config is **always read on startup** (and on demand via a "Resync" button — no container restart required):

- **New services** (not in DB by name): imported and stored to DB
- **Existing services** (name already in DB): skipped silently
- **Duplicate entries** within the JSON file itself, or a name collision where JSON data differs from DB data: surface a warning notification in the UI and log to a dedicated **Warnings & Errors tab**
- **No JSON / empty JSON**: nothing imported; user adds services manually via UI
- **UI-created services**: stored to DB only — never written back to JSON file
- The JSON file is an additive, one-way import feed — DB is always the runtime source of truth

**Key design element:** Dedicated "Warnings & Errors" tab in the UI for surfacing config mismatches, import conflicts, and other system-level alerts. Alarmistic — draws attention.

### Authentication (FINAL)

- **Username + password login** — standard form
- **JWT token** generated on successful login, stored in browser (httpOnly cookie or localStorage — implementation decision, cookie preferred for security)
- Token is long-lived; user stays logged in across sessions
- **Auto-registration on first login:** if username does not exist in DB → account created automatically → onboarding popup guides user through app basics
- No admin pre-provisioning required; first user in is effectively the owner
- Auth disabled via env var (`FISHTANK_AUTH_ENABLED=false`) for local dev convenience

**Open question logged:** With auto-registration, any person who can reach the network can create an account. Given "internal network only" scope this is acceptable — but should there be a toggle to disable auto-registration (invite-only mode) for tighter shared environments? Low priority, flag for later.

**RESOLVED:** Auto-registration toggle confirmed as a feature — `Auth` settings will include an `AllowAutoRegistration` flag (default: true). Invite-only mode is in scope.

### Frontend Stack (FINAL)

**React (TypeScript)** — chosen for speed, ecosystem maturity, and charting/graph capability (Recharts, Victory, etc.) for future analytics features. Runs as a SPA served as static files, fully decoupled from the .NET backend. No Blazor/Angular.

### Database (FINAL)

**SQLite via EF Core** — single file, volume-mountable, zero external process.

**What IS stored in DB:** services config, users, auth tokens, network activity/request log, user actions, audit info, warnings/errors, system notifications.

**What is NOT stored in DB:** mappings and response files — these live on the filesystem only.

### Mappings & Responses — Filesystem Architecture (FINAL)

Mappings and responses are **filesystem-only** — never in the DB.

**Canonical folder structure (inside container):**
```
mocks/
  services/
    <service_name>/
      mappings/
      responses/
```

This folder is **volume-mounted** by the end user in their Dockerfile/docker-compose — identical structure must be mirrored in their project. This is the same pattern as the existing `conf/services/<name>/` structure from the config example, normalized.

**Two-way sync model:**
- UI creates/edits mapping → written to filesystem immediately → reflected on disk on the user's machine (via volume mount)
- User creates files manually on disk → "Resync Mappings" button in UI → files loaded into UI display AND mock server restarts to pick up new mappings
- DB records **user actions only** (who created/edited/deleted what, when) — not the mapping content itself

**Key implication:** The mock server must support hot-restart or reload when mappings change. WireMock.NET supports this via its admin API (`POST /__admin/mappings/reset`). A full process restart may also be acceptable given the "enable/disable on the fly" requirement already implies restarts.

**RESOLVED — Restart scope:** Only the **WireMock.NET process restarts**, not the container. The ASP.NET backend (API + UI serving) stays alive. The UI shows a brief "Mock server restarting..." state and reconnects cleanly. This gives fast reload without container downtime.

### Feature Depth — v1 Confirmed

**Mock Suggestions (FINAL — three interaction modes):**
1. **Per-row action button** — every proxied request row in the network activity list shows a "Save as Mock" button
2. **Detail popup** — clicking a proxied request opens a popup showing the full request + response with a "Save" button that creates the WireMock mapping + response file on disk
3. **Record mode toggle** — a toggle on the network activity tab that, when ON, automatically promotes every proxied request to a mapping + response file as it arrives (bulk recording mode)
All three modes write files to the filesystem (the volume-mounted mappings/responses structure). No AI inference — clean, deterministic, one-to-one capture.

**Admin Console (FINAL — full scope):**
- User management (create, edit, deactivate, invite)
- Feature toggle management (on/off per feature, with env var override visibility)
- System metrics (mock server status, request throughput, error rates)
- Health dashboard (service health, container uptime, WireMock.NET process status)
- Audit log viewer (who did what, when — sourced from DB audit table)
- Warnings & Errors viewer (config mismatches, import conflicts, system alerts)

### V1 Scope Confirmation

**Everything discussed is v1.** No deferred features. Depth confirmed above for the two ambiguous areas.

### Feature Toggles (FINAL)

- **Stored in DB** — toggles are persisted, not file-based
- **Scope:** primarily UI features, backend features where applicable
- **Default state:** all features ON in every release
- **Override via env vars** — e.g. `FISHTANK_FEATURE_MOCKSUGGESTIONS=OFF`; end user sets these when pulling the image to disable unwanted features or roll back a broken feature fast
- **Purpose:** strip down for simpler use cases AND emergency rollback of a bad release without redeployment
- This is a first-class deployment configuration mechanism, not a developer debug flag

### Authentication — Admin User & Token Longevity (FINAL)

**Default admin account (FINAL):**
- Username: `admin` — always fixed
- Password: resolved at startup in this order:
  1. `FISHTANK_ADMIN_PASSWORD` env var → used as-is, no forced change (explicit configuration)
  2. No env var → default `admin`, **forced password change on first login before any access** — no skip, no dismiss
- Admin account has access to an **Admin Console** (user management, feature toggles, system settings, invite controls)

**Token longevity:**
- JWT is **very long-lived** — expires only when the container restarts (no arbitrary time expiry)
- On container restart: all sessions invalidated, users must log in again
- Rationale: internal-network only, acceptable risk; user experience wins over session churn
- ⚠️ **Security flag logged:** no token expiry means a stolen token is valid indefinitely until restart. Acceptable given org-network boundary, but worth documenting as a known trade-off in the README.
